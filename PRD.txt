# MIRAI NIKKI

## Product Requirements Document (PRD)

### Version 1.0

---

# 1. Overview

Mirai Nikki is a privacy-first, AI-assisted, mobile-first Progressive Web Application that helps users commit to tomorrow by writing tomorrow's achievements as though they have already happened.

Unlike traditional productivity applications that ask users to plan tomorrow using lists and reminders, Mirai Nikki asks users to narrate tomorrow in the past tense.

Example:

> "Today I woke up at 6 AM, completed the authentication module, called my parents, watered the plants and finished reading my book."

The application then uses an LLM to transform this narrative into a structured execution plan.

The original writing is sealed immediately after submission and cannot be viewed until the following morning.

When the user opens the application the next day, they first read their "future memory", followed by a checklist generated from that memory.

The application's primary purpose is **commitment**, not productivity.

Artificial Intelligence is intentionally invisible. Users should never feel like they are interacting with an AI assistant. AI exists solely to convert natural language into structured actions.

---

# 2. Product Goals

The product should:

* Encourage intentional planning through narrative writing.
* Reduce morning decision fatigue.
* Make AI invisible to the user.
* Feel like a daily ritual rather than a productivity tool.
* Operate entirely without proprietary backend infrastructure.
* Keep all user data on-device.
* Function as an installable Progressive Web Application.

The product should NOT attempt to become:

* A second brain
* A note-taking application
* A project management tool
* A calendar
* A habit tracker
* A chatbot
* A therapist
* A social platform

Every feature request should answer one question:

> Does this strengthen the nightly commitment ritual?

If the answer is no, it should not be implemented.

# 12. Task Execution

Tasks are extracted by the Future Compiler.

Each task contains

* Title
* Completion state
* Optional category
* Optional estimated time

Completing a task

* Marks checkbox complete
* Leaves task visible
* Slightly reduces opacity
* Adds strike-through
* Triggers corresponding sentence highlight if mapping exists

Completed tasks are never removed.

History should remain honest.

Tasks cannot be reordered manually.

The AI decides the initial ordering based on dependency and chronology.
# 13. Archive

The Archive represents completed days, not completed tasks. It should feel like browsing chapters of a book rather than browsing historical data.

Entries are grouped by month and sorted newest first. Each entry is displayed as a hardcover journal card containing the date, day of week, and one status badge.

Status values are intentionally limited to:

* Fulfilled
* Left Unfinished

Do not display percentages, streaks, productivity scores, completion rates or achievement badges. The archive is intended to preserve history, not judge it.

Opening an archived entry displays:

* Original future memory
* Timestamp written
* Unlock timestamp
* Compiled execution plan
* Completion state of every task
* Notes indicating if AI compilation failed or required retry

Archived entries are read-only.

Editing is never permitted after sealing.

Deletion must only be possible from Settings and always require explicit confirmation.

# 16. Future Compiler

The AI module is internally called the Future Compiler.

Never refer to it as "Chat", "Assistant" or "Parser".

Responsibilities:

* Convert narrative into structured tasks.
* Infer obvious prerequisite tasks only when confidence is high.
* Preserve chronological order where possible.
* Return strict JSON.
* Never rewrite the journal.
* Never summarize the journal.
* Never provide advice.
* Never motivate.
* Never comment on the user's writing.

Example:

Input:

"Today I finally submitted my college application."

Output:

* Finish application
* Upload documents
* Submit application

If confidence is low, prefer fewer tasks rather than inventing work.

---

# 17. Future Compiler Prompt

The default system prompt should instruct the model to behave as a deterministic compiler.

Requirements:

* Extract only actionable tasks.
* Ignore emotional language.
* Ignore compliments, opinions and storytelling.
* Do not rewrite user text.
* Return JSON only.
* Never return markdown.
* Never include explanations.
* Preserve ordering when possible.
* Merge duplicate tasks.
* Infer dependencies conservatively.

Application-side JSON validation is mandatory.

If validation fails:

1. Retry once.
2. Attempt lightweight repair.
3. If still invalid, preserve the journal and mark compilation as failed.

# 19. Local Storage

Use IndexedDB via Dexie.

Storage responsibilities:

Journal Table
Stores every sealed and archived entry.

Task Table
Stores compiled execution plans.

Draft Table
Stores active writing session.

Settings Table
Stores application configuration.

Queue Table
Stores pending AI compilation requests.

Metadata Table
Stores application state.

Never store API keys inside exported backups unless the user explicitly opts in.

# 20. Offline Behaviour

The application must remain fully usable without internet.

Available offline:

* Read archive
* Write tomorrow
* Edit draft
* Change settings
* Complete tasks
* Export data
* Import data

Unavailable offline:

* AI compilation
* AI connection test
* Provider model listing

If a journal is sealed while offline:

* Seal immediately.
* Save locally.
* Add to compilation queue.
* Retry automatically when connectivity returns.

If compilation has not completed by unlock time:

Display the journal normally.

Replace the task list with:

"Your execution plan is still being prepared."

Provide a Retry button.

The journal is never blocked by AI.

# 24. Error Handling

AI Failure

* Preserve journal.
* Mark compilation pending.
* Allow retry.

Invalid API Key

* Explain clearly.
* Never expose raw HTTP errors.

Malformed AI Response

* Retry once.
* Attempt repair.
* Preserve journal.

IndexedDB Failure

* Display recovery instructions.
* Offer export if possible.

Import Failure

* Validate schema before overwrite.
* Reject invalid backups.

Application updates must preserve all stored data.

* Custom splash screen
* App shortcuts
* Offline asset caching
* Versioned cache management
* Background update detection
* Automatic cache invalidation on new releases

# 30. Edge Cases

User closes browser while writing.

Expected behavior:

Draft restored automatically.

---

User refreshes page during AI compilation.

Expected behavior:

Compilation resumes or retries.

Journal never duplicated.

---

User changes unlock time after sealing.

Expected behavior:

Previously sealed entries retain their original unlock time.

Only future entries use the new setting.

---

User changes AI provider.

Expected behavior:

Future entries use new provider.

Historical entries remain unchanged.

---

User imports backup over existing data.

Prompt:

Replace existing data?

Merge with existing data?

Cancel

---

System clock changes.

Use the device's current local time.

No attempt should be made to prevent deliberate manipulation.

Mirai Nikki relies on the user's commitment rather than enforcing behavior through technical restrictions.

# 36. Guiding Principles for Implementation

When making implementation decisions, prioritize simplicity over feature count.

Do not add features because other productivity applications have them.

The interface should remain quiet, uncluttered and predictable.

Animations should communicate state changes rather than entertain.

Artificial Intelligence should remain invisible. Users should never feel like they are using an AI application. They should feel like they are writing in a diary that quietly prepares tomorrow for them.

The writing experience is the most important part of the application. The AI, checklist and archive exist to support that experience, not replace it.

If two design options are equally functional, choose the one with fewer controls, less visual noise and lower cognitive load.

Every screen should reinforce the product's central idea: the person writing tonight is making a promise to the person waking up tomorrow.

---

# 37. Final Product Statement

Mirai Nikki is a local-first, privacy-first Progressive Web Application that transforms tomorrow's imagined memories into today's executable commitments.

Users write tomorrow as though it has already happened. The application silently compiles that narrative into a structured execution plan, seals it until morning, and then reveals it as a letter from yesterday's self.

The application is intentionally minimal, avoids unnecessary productivity mechanics, requires no backend infrastructure, and uses AI only where it provides clear functional value.

The goal is not to help users plan more.

The goal is to help them keep the promises they already made to themselves.
